# 電話番号の補完（Google Places ＋ 確認キュー）

厚労省オープンデータには**電話番号が含まれない**ため、別途補完します。
誤った番号は「違う医院に架電」という実害につながるので、**機械は候補出し・人が最終判断**する
「信頼度つき＋確認キュー」方式を採用しています。

## 仕組み（誤マッチ防止）

1. `phone` 未設定の医院について Google Places を「名称＋住所」で検索
2. 各候補を **名称類似度（bigram Dice）＋ 厚労省座標との距離（haversine）** でスコアリング
   （座標一致が強い裏付けになり、同名でも遠い別医院を弾ける）
3. 判定（`lib/match.ts` の `decideMatch`）:
   - **高信頼**（名称一致≥0.8 ・ 距離≤150m ・ 2位と十分差）→ **自動採用**（ただし `phone_verified=false`＝未確認）
   - **曖昧** → `phone_candidates`（確認キュー）に積む
   - 候補なし／電話なし → 何もしない
4. 採用・未確認の番号は UI で「**未確認**」バッジ表示。架電者が確認 → 確認済みに
5. **手入力・確認済みの番号は自動処理で上書きしない**。`place_id`・距離・スコアも記録（監査可能）

| 入手経路 | `phone_source` | `phone_verified` |
| --- | --- | --- |
| 手入力（詳細画面） | `manual` | `true` |
| 確認キューで人が承認 | `places` | `true` |
| 自動採用（高信頼） | `places` | `false`（未確認） |

## しきい値

`lib/match.ts` の `THRESHOLDS` で調整できます（自動採用の信頼度・距離・2位との差など）。

## 使い方

### 1. APIキー無しで照合ロジックを確認（おすすめの最初の一歩）

```bash
pnpm enrich:phones --mock scripts/__fixtures__/places.sample.json --dry-run
```

サンプルには「正解／やや違い／同名だが遠い別物／電話なし」が入っており、
**遠い別物が自動採用されず確認キュー行きになる**ことを確認できます。

### 2. 本番（Google Places で補完）

必要な環境変数:
- `NEXT_PUBLIC_SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`
- `GOOGLE_MAPS_API_KEY`（Places API (New) を有効化したキー）

```bash
# まず dry-run（書き込みなし・判定だけ表示）
NEXT_PUBLIC_SUPABASE_URL=... SUPABASE_SERVICE_ROLE_KEY=... GOOGLE_MAPS_API_KEY=... \
  pnpm enrich:phones --prefecture 東京都 --city 新宿区 --limit 30 --dry-run

# 本実行
NEXT_PUBLIC_SUPABASE_URL=... SUPABASE_SERVICE_ROLE_KEY=... GOOGLE_MAPS_API_KEY=... \
  pnpm enrich:phones --prefecture 東京都 --city 新宿区 --limit 30
```

### 3. 確認キューを人がさばく

アプリの **`/clinics/review`**（下タブ「番号確認」）で、自動採用されなかった候補を
信頼度・距離つきで確認し、正しい番号を「採用」または「全部違う」で処理します。

## オプション

| フラグ | 説明 |
| --- | --- |
| `--prefecture` / `--city` | 対象の絞り込み |
| `--limit N` | 処理件数の上限（既定50） |
| `--max-candidates N` | 1医院あたりの候補数（既定5） |
| `--mock <json>` | APIの代わりにローカルJSONを使う（検証用） |
| `--dry-run` | DB書き込みせず判定のみ表示 |

## コスト・規約の注意

- Places API (New) の電話番号取得は課金対象。`--prefecture/--city/--limit` で範囲を絞る運用推奨。
- Google の規約上、長期保存が許容されるのは基本的に `place_id` のみ。電話番号は
  「未確認」を起点に**人が確認して自社の正データに昇格**させる運用（本設計）が安全。
